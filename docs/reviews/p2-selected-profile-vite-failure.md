## Remaining blocking finding

### P2-V05 — A validated exited-state outcome is dropped on cross-check failure

`executeFixedViteLifecycleAttempt()` parses the final inspect and successfully
applies `requireInspect(..., "exited")`, establishing the owned container's
fixed exited-state outcome (`src/vite-executor.ts:1230-1232`). It then checks
the final image against the created-state image and the Docker start exit code
against the inspected exit code (`src/vite-executor.ts:1233-1238`). Only after
both comparisons pass does it assign `inspectedContainerExitCode`
(`src/vite-executor.ts:1239`). `buildAttemptRecord()` later consumes that
nullable field directly (`src/vite-executor.ts:1369-1370`).

A Docker-free fake-command reproduction supplied one owned container, one fixed
image, a valid final `exited|1` inspect, and a Docker start result with exit 0.
The cross-check correctly made the attempt Inconclusive, but the canonical
attempt projection retained the created-state image ID and emitted
`inspectedContainerExitCode: null` rather than the established inspected value
`1`. The 112-test suite covers unknown start settlement, malformed runner
framing after a matching final inspect, and early create/inspect failures, but
not this intervening cross-check transition.

Impact: the attempt record still cannot distinguish “final inspect not
established” from “final inspect established, then cross-binding failed.” This
is the exact diagnostic-state loss P2-V05 was intended to remove. It does not
authorize evidence access or a receipt, but it blocks closure and any new-run-ID
execution gate.

Required remediation:

1. Store the inspected container exit code immediately after the fixed final
   `exited` inspect validates, before image/start-exit cross-checking.
2. Add fake-command-to-attempt-record regressions for a start/inspect exit-code
   mismatch and an after/before image mismatch. Both must retain the exact
   validated partial fields, keep evidence untouched, and remain Inconclusive.
3. Rebuild the exact staging tree and submit the implementation/test/staging
   bytes to another fresh Docker-non-executing re-review.

## Blocking findings

### P2-V05 — The attempt record drops already-inspected lifecycle state

`executeFixedViteLifecycleAttempt()` validates the first `created` inspect but
stores its image identity only in a block-local `before` value. It assigns
`lifecycle` only after attached start, the final `exited` inspect, and runner
framing all validate (`src/vite-executor.ts:1135-1150`). If start settlement is
unknown, the record therefore loses the already-inspected image ID. If the
final inspect succeeds but runner framing is malformed, it also loses the
already-inspected container exit code.

`buildAttemptRecord()` obtains both nullable fields only from the all-or-nothing
`lifecycle` object (`src/vite-executor.ts:1225-1287`), so those later failures
write `null` even though the executor established a bounded value before the
failure. The lifecycle tests assert only that such paths reject; the
finalization tests begin with a fully synthesized lifecycle and never connect a
fake command transition to the canonical attempt record.

Impact: the new record can still discard the exact inspected image/container
outcome it was introduced to preserve. `null` cannot distinguish “not reached”
from “established, then lost,” weakening failure diagnosis and any later
same-image/new-gate review.

### P2-V06 — Production evidence access is non-atomic and its export path is not behaviorally covered

`readFixedEvidence()` starts the bounded event read and both metadata checks in
one `Promise.all` (`src/vite-executor.ts:961-980`). If one output path is
inaccessible, another branch may already have opened and read the event
segment. `finalizeFixedViteExecution()` then reports the entire outcome as
`evidence: "not-inspected"` with `P2_RECEIPT_ASSEMBLY_FAILED`
(`src/vite-executor.ts:1329-1347`). The recorded classification can therefore
contradict the operations that actually occurred.

The “exported fixed paths” test injects already-constructed evidence, and the
inaccessible-metadata test injects an atomic backend rejection
(`test/vite-executor.test.ts:462-539`). The runner lifecycle test injects a
synthetic `verifyOutput()` function, while the only check of the production
`verifyFixedOutput()` chmod boundary is a source-string assertion
(`test/vite-runner.test.ts:170-187`). No Docker-free behavioral test exercises
the real fixed-path validation/export modes followed by the production bounded
reader, or proves that an inaccessible path cannot be mislabeled after a
partial read.

Impact: the explicit P2-V04 acceptance requirement for a successful receipt
through exported fixed paths and truthful inaccessible-output handling is not
met. A regression in the production modes, exact path set, or availability
ordering can pass all 105 tests.

## Blocking finding

### P2-V04 — Receipt assembly crosses unavailable output and loses attempt state

`executeFixedViteLifecycle()` retains a parsed runner disposition and suppresses
container cleanup when runner settlement is unknown
(`src/vite-executor.ts:998-1048`). `executeOne()` then unconditionally calls
`buildReceipt()` (`src/vite-executor.ts:1058-1065`). The receipt builder opens
the event segment and traverses the tool output path in one `Promise.all`
(`src/vite-executor.ts:878-902`) before writing `summary.json`.

That access shape is incompatible with the runner's retained-output boundary:

- unknown runner settlement intentionally suppresses event chmod, leaving a
  container-owned partial segment unavailable to the host;
- `prepareFixedInputs()` always creates `tool/out` as container-owned mode
  `0700` (`runner/vite-runner.js:196-215`), so the host cannot independently
  traverse it unless a known-settled success path explicitly exports it; and
- `readBoundedEventFile()` and `optionalRegularFile()` collapse inaccessible
  output into `P2_EXECUTOR_OUTPUT_INVALID` rather than a bounded
  `not-inspected` evidence state.

When receipt assembly rejects, the entry catches the error and emits only
`P2_EXECUTOR_FAILED` (`runner/vite-executor-entry.js:53-57`). The already parsed
runner disposition, image identity, container exit, and cleanup decision remain
only in process memory, and `summary.json` is never written. Existing tests
exercise lifecycle cleanup suppression but do not run receipt assembly against
inaccessible event/output paths.

Impact: a safely retained settlement-unknown result and even a known-settled
runner result can fail before a canonical attempt record exists. The executor
also attempts to consume evidence that the runner deliberately left
unavailable. A generic entry failure cannot distinguish those paths afterward,
so neither profile evidence nor a safe retry decision can be derived from the
retained tree.

## Bounded remediation contract

A Docker-non-executing remediation is possible. It must satisfy all of the
following before any new execution gate is proposed:

1. Persist a small canonical **attempt record** from the fixed lifecycle state
   before opening or traversing container-owned evidence paths. Keep it
   distinct from a complete selected-profile receipt and retain only fixed
   scenario/profile/run identity, nullable inspected image/container outcome,
   sanitized runner disposition, Docker/runner settlement, cleanup disposition,
   and allowlisted issue codes.
2. Represent unavailable evidence explicitly as `not-inspected`; do not encode
   it as absent or zero. If Docker or runner settlement is unknown, do not open,
   stat through, enumerate, or chmod the event, direct-write, or tool-output
   subtrees. The attempt remains `inconclusive`, and no `summary.json` candidate
   receipt is created.
3. Establish a host-readable output boundary only after known settlement. A
   successful runner must validate the exact event segment and single fixed
   entry output, then expose only those fixed paths with reviewed modes before
   the executor independently performs bounded reads. Known failures may retain
   evidence as `not-inspected`; they must not require output traversal merely to
   preserve the attempt record.
4. Keep the entry projection bounded while preserving whether the canonical
   attempt record was written. It must not collapse record-write failure and a
   recorded inconclusive attempt into the same success-like state.
5. Add Docker-free behavioral coverage for a successful receipt through the
   exported fixed paths, a known runner failure, an unknown-settlement path whose

   attempt-record write failure, and the rule that only two complete receipts
